  TL;DR on your three questions

  RLHF = the original recipe: SFT → train a reward model on human preference
  pairs → optimize policy with PPO + KL-to-reference. Subjective alignment,
  expensive labels, reward-model drift.

  RLVR = same idea, swap the learned reward model for a programmatic verifier
  (math answer-check, code unit tests, proof checkers, format regex). This is
  what produced DeepSeek-R1, Tülu 3, and the modern "reasoning model" wave. RLVR
   is orthogonal to the algorithm — PPO, GRPO, DAPO, GSPO can all be RLVR; it's
  a statement about reward source.

  Is GRPO still the leader? It's still the default baseline but the frontier has
   moved one step further:
  - GRPO (DeepSeek-Math, 2024) — drop the value model, use group-relative
  z-score advantages. Best general default.
  - DAPO (ByteDance, early 2025) — GRPO + four patches: Clip-Higher (asymmetric
  ε), token-level loss aggregation, dynamic sampling, overlong-reward shaping.
  Better for long CoT.
  - GSPO (Qwen team, 2025; used to train Qwen3) — importance ratio is
  sequence-level, length-normalized. The right answer for MoE models and very
  long sequences, where per-token importance sampling is too noisy.

  Recent comparative work (arxiv 2512.07611) suggests on dense math reasoning,
  GRPO/DAPO/PPO end up close, with DAPO's Clip-Higher + token-level loss being
  the parts that actually matter. TRL v1.0's GRPOTrainer exposes all three as a
  loss_type flag, so A/B'ing them is one line.

  Other approaches worth knowing: DPO/SimPO/KTO/ORPO (closed-form, no rollout
  loop — cheap and stable but can't generate new reasoning behaviors), RLAIF
  (LLM-judge labels), self-play (SPIN, SPICE), agentic RL (verl-agent, NeMo
  Gym).

  On Qwen3.5 / Qwen3.6

  Both exist:
  - Qwen3.5 (Feb 2026) — flagship MoE (397B-A17B, 122B-A10B, 35B-A3B) plus a 
  Small series at 0.8B / 2B / 4B / 9B. The Small series is the sweet spot for
  this project.
  - Qwen3.6 (Apr 2026) — currently 27B dense + 35B-A3B MoE, Apache-2.0; more
  sizes "coming."

  For learning + iterating, I'd start at Qwen3.5-0.8B base (full-parameter GRPO
  on one GPU), then scale.

  What's compatible: anything decoder-only with logprobs exposed in transformers
   — Llama, Gemma, Qwen, Mistral, Phi, SmolLM, DeepSeek, etc. Incompatible:
  encoder-only (BERT), vision-only models, anything without exposed logprobs.
  The only model-family-specific concerns are chat template, attention impl, and
   whether it's MoE (which steers you toward GSPO).

  Approach options

  Path: Pure PyTorch from scratch
  Best for: Understanding every line
  Tradeoff: ~150 LOC for GRPO; you own all the plumbing (KV cache, ref-model
    memory, weight sync)
  ────────────────────────────────────────
  Path: TRL v1.0
  Best for: Fast path to working GRPO/DAPO/GSPO + DPO/SFT in one stack
  Tradeoff: Some choices baked in
  ────────────────────────────────────────
  Path: Unsloth
  Best for: Max VRAM efficiency on 1-2 GPUs (~80% less VRAM, ~2x speed)
  Tradeoff: Single-node only
  ────────────────────────────────────────
  Path: verl (already cloned at /home/djole/code/verl)
  Best for: Serious research, multi-node, 200+ knobs
  Tradeoff: Bigger learning curve
  ────────────────────────────────────────
  Path: OpenRLHF / NeMo-Aligner
  Best for: Production multi-node RLHF
  Tradeoff: Only needed if scaling

  My suggested order: pure PyTorch on a tiny model first to internalize the
  algorithm, then TRL or Unsloth for real runs, then verl if/when we need scale.

  What pure-PyTorch GRPO looks like

  The whole inner loop is genuinely ~150 lines. Sample G completions per prompt
  → score with verifier → compute (r - mean)/std group-relative advantage →
  clipped PPO-style update with KL to a frozen reference. DAPO = three small
  patches to that loop. GSPO = replace the per-token ratio with a
  length-normalized sequence-level one. Full skeleton with PyTorch is in Section
   4 of PROJECT_NOTES.md.

  The hard parts are not the algorithm — they're: keeping policy + frozen-ref +
  optimizer-state in memory, fast sampling (vLLM vs .generate), tokenization
  alignment for logprobs, and distributed training. Which is why people use
  TRL/verl.
